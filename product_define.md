# 研发与质量一体化管理系统开发任务书

## 1. 项目概述

### 1.1 项目背景
公司目前通过多个Excel表格管理研发项目与质量问题，存在以下痛点：
- **历史记忆丢失**：已完成项目/问题的记录经常被删除，导致相同问题重复发生
- **协作效率低下**：跨部门协作依赖文件传输，版本混乱
- **状态跟踪困难**：无法实时了解各项事务进展
- **数据孤岛**：研发、质量、售后信息分散，难以统一分析

### 1.2 项目目标
构建一套**轻量级、易用、保留完整历史**的研发与质量一体化管理系统，取代当前的Excel管理方式。

### 1.3 核心设计原则
1. **不增加额外工作量**：系统使用门槛不高于Excel
2. **历史可追溯**：禁止物理删除，所有操作留痕
3. **状态驱动而非流程驱动**：适应非标准化的研发工作
4. **渐进式优化**：先解决核心痛点，后期可扩展

## 2. 核心需求

### 2.1 必做功能（MVP）

#### 2.1.1 事项统一管理
- 将研发任务、质量改进、售后问题统一为"事项"进行管理
- 支持事项的基本属性记录：标题、描述、关联项目、处理人等
- 事项分类：问题修复/质量改进/新功能/优化任务

#### 2.1.2 状态流转看板
- 提供看板视图（Kanban Board），支持拖拽操作
- 默认状态流程：待处理 → 处理中 → 待验证 → 已完成 → 已归档
- 状态切换时记录操作人和时间

#### 2.1.3 时间自动记录
- 创建时间、最后更新时间自动生成，无需手动填写
- 支持按时间线查看事项历史
- 延期自动标记（基于计划完成时间）

#### 2.1.4 审计追踪
- 所有字段变更自动记录：谁、何时、改了什么
- 变更记录可追溯，形成完整时间轴
- 物理删除功能禁用（仅管理员可逻辑归档）

#### 2.1.5 附件管理
- 支持上传图片、文档、视频等作为处理证据
- 附件与事项关联，可在线预览
- 附件存储空间管理

### 2.2 扩展功能（后续迭代）

#### 2.2.1 项目维度
- 项目主页，聚合相关所有事项
- 项目进度概览（基于事项完成情况）
- 项目文档集中管理

#### 2.2.2 统计分析
- 事项按部门、类型、处理人统计
- 处理时长分析（平均处理时间、最长处理时间）
- 延期事项分析

#### 2.2.3 通知提醒
- 事项分配给处理人时自动通知
- 即将到期事项提醒
- 重要事项状态变更通知

#### 2.2.4 移动端适配
- 移动端查看、处理事项
- 移动端拍照上传附件
- 移动端接收通知

## 3. 数据模型设计

### 3.1 核心实体

#### 3.1.1 用户（User）
```javascript
{
  id: "用户唯一ID",
  name: "姓名",
  department: "部门", // 销售/售后/技术/生产
  avatar: "头像URL",
  isActive: "是否激活"
}
```

#### 3.1.2 项目（Project）
```javascript
{
  id: "项目编号", // 如PZ-2025120801
  name: "项目名称", // 如"天裕"
  type: "项目类型", // R&D/质量提升/售后支持
  priority: "优先级", // 1-5
  status: "项目状态", // 规划中/进行中/已完成/已暂停
  startDate: "开始日期",
  targetDate: "目标完成日期",
  archived: "是否已归档" // 禁止删除，只归档
}
```

#### 3.1.3 事项（Issue）- 核心表
```javascript
{
  id: "事项编号", // 自动生成 ISS-2025-001
  title: "标题", // 如"2D视觉抓取稳定性提升"
  description: "详细描述",
  projectId: "关联项目ID", // 可选
  
  // 分类信息
  type: "事项类型", // 问题/任务/需求
  category: "问题类别", // ["调试","硬件","软件BUG"]
  equipmentType: "设备类型", // ["关节","2D视觉"]
  severity: "严重程度", // 致命/严重/一般/轻微
  
  // 责任人
  reporterId: "提出人ID", // 谁发现的
  assigneeId: "处理人ID", // 谁负责处理
  relatedPersonIds: ["相关人员IDs"], // 相关协作人员
  
  // 状态跟踪
  status: "当前状态", // 待处理/处理中/待验证/已完成/已归档
  progress: "进度", // 0-100 百分比
  isBlocked: "是否受阻", // 受阻标记
  blockReason: "受阻原因",
  
  // 时间管理
  createdAt: "创建时间", // 自动
  updatedAt: "最后更新时间", // 自动
  plannedStart: "计划开始时间", // 可选
  plannedEnd: "计划完成时间", // 可选
  actualStart: "实际开始时间", // 自动/手动
  actualEnd: "实际完成时间", // 自动/手动
  
  // 自动计算
  isDelayed: "是否延期", // 自动计算
  delayDays: "延期天数", // 自动计算
  
  // 关联
  parentId: "父事项ID", // 支持事项拆分
  tags: ["标签"], // 自定义标签
  attachments: ["附件URLs"], // 附件列表
}
```

#### 3.1.4 审计日志（AuditLog）
```javascript
{
  id: "日志ID",
  issueId: "关联事项ID",
  userId: "操作用户ID",
  action: "操作类型", // CREATE/UPDATE_STATUS/UPDATE_ASSIGNEE/ADD_COMMENT/ADD_ATTACHMENT
  timestamp: "操作时间", // 自动
  fromValue: "变更前值", // JSON字符串
  toValue: "变更后值", // JSON字符串
  comment: "备注", // 操作时的备注
  ipAddress: "操作IP"
}
```

#### 3.1.5 评论/备注（Comment）
```javascript
{
  id: "评论ID",
  issueId: "关联事项ID",
  userId: "评论人ID",
  content: "评论内容", // 支持富文本
  attachments: ["附件"], // 评论附件
  isInternal: "是否内部备注", // 区分对外回复
  createdAt: "评论时间"
}
```

### 3.2 数据关系
```
用户(User) 1:n 事项(Issue) ←---→ 审计日志(AuditLog)
    ↑               ↑                     ↑
    |               |                     |
    |               |                     |
项目(Project) 1:n 事项(Issue) ←---→ 评论(Comment)
    |               |
    |               |
    |               n
    |             附件(Attachment)
    |
    |
 标签(Tag) n:n 事项(Issue)
```

## 5. 技术规格建议

### 5.1 非功能性需求

#### 5.1.1 性能要求
- 页面加载时间 < 2秒
- 看板拖拽响应时间 < 200ms
- 支持100人并发使用
- 支持10,000+事项存储

#### 5.1.2 安全性
- 企业账号集成登录
- 操作日志完整记录
- 附件访问权限控制
- 数据备份与恢复

#### 5.1.3 可用性
- 系统可用性 > 99.5%
- 支持离线数据暂存（网络恢复后同步）
- 浏览器兼容：Chrome 80+、Edge 80+、Safari 13+

### 5.2 部署架构建议
```
前端(SPA) → API网关 → 后端服务 → 数据库
   │           │           │         │
   │           │           ├─Redis(缓存)
   │           │           │
   │           │           ├─MinIO/S3(附件存储)
   │           │           │
   │           │           └─Elasticsearch(搜索)
   │           │
   └─WebSocket←─┘  (实时通知)
```

### 5.3 技术栈建议（可选）
- 前端：Vue 3 + Element Plus 或 React + Ant Design
- 后端：Node.js + NestJS 或 Java + Spring Boot
- 数据库：PostgreSQL（主库） + Redis（缓存）
- 搜索：Elasticsearch
- 存储：MinIO（自建）或 阿里云OSS
- 部署：Docker + Kubernetes 或 传统服务器部署

## 7. 成功标准

### 7.1 上线标准
- 现有Excel数据可完整导入
- 核心团队（10人）完成培训并日常使用
- 系统稳定性达到99.5%
- 响应时间满足性能要求

### 7.2 验收标准
- 90%以上的新事项通过系统创建
- 历史事项查询时间缩短50%
- 跨部门协作沟通时间缩短30%
- 零数据丢失（相比Excel的随意删除）

### 7.3 衡量指标
- 用户活跃度：DAU/MAU
- 事项处理平均时长
- 用户满意度调查（每月）
- 系统错误率
