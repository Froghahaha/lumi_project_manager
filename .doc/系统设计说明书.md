# 项目管理系统设计说明书

## 问题定义
当前的项目管理，总共依赖3个人员维护各自的excel表格。这使得数据存在冗余。

## 总体流程
机械设计 ──→ 生产 ──→ 调机 ──→ 验收 ──→ 尾款
    │          │        │
    └── 软件设计 ────────┘
        (横跨这三个阶段)

## 项目管理流程设计构想
本系统设计了以下流程管理项目，对于单个项目来讲，流程是线性的，顺序如下
1. 创建项目工单
销售助理在收到合同后，创建项目。自动生成项目基础信息（项目order_no、客户信息、合同签订日期、交期）
2. 技术主管指派人员
技术主管指派项目经理、机械生产执行人、机械设计执行人、软件设计执行人、生产执行人
3. 机械设计人员维护设计阶段状态
未开始→设计中→图纸已下发 这几个阶段完成后，机械设计完成
4. 生产执行人维护生产阶段状态
未开始→生产中→生产完成→已发货 这几个阶段完成后，生产完成
5. 售后主管指派人员
售后主管指派安调执行人
6. 安调执行人维护安调阶段状态
未开始→安调中→安调完成 这几个阶段完成后，安调完成
7. 验收
售后主管按照实际线下信息维护验收状态
8. 收款进度实时更新
销售助理按照实际线下信息维护收款进度

## 分角色工作台
1. 项目经理工作台
   项目经理负责项目管理，其工作台主要职责是观看各项目进度，处理项目问题。其可以看到其被指派的所有项目信息，同时辅助其按照项目交期、项目节点预警等信息进行项目管理，并询问/记录项目各流程中问题。
2. 机械设计执行人工作台
    机械设计执行人负责机械设计，其工作台主要职责是查看被指派到自己头上的设计任务，按照设计要求进行设计，并维护设计状态，a→b→c 这几个阶段完成后，机械设计完成。
3. 生产执行人工作台
    机械生产执行人负责机械生产，其工作台主要职责是查看被指派到自己头上的生产任务，按照生产要求进行生产。其应该可以看到项目中前一个环节（机械设计）的完成状态。首先其首要优先级是进入生产环节的项目，其此是辅助其查看其他未进入生产环节的项目的机械设计进度，方便其及时调整生产计划。
4. 软件设计执行人工作台
    与项目经理看板大致相同，辅助其规划软件设计时间。
5. 售后主管工作台
    其工作是借助看板，掌握各个进入生产环节的项目的发货状态，并根据情况指派安调执行人。
7. 销售助理工作台
    - 在工作台中依据线下签订的合同，创建项目工单，录入项目基础信息。
    - 维护项目收款进度。
8. 技术主管工作台
    - 为项目指派项目经理、机械生产执行人、机械设计执行人、软件设计执行人、生产执行人
9. 超级管理员工作台
    - 管理所有用户账号，包括普通用户、管理员账号。
    - 管理所有项目工单，有一个项目总览视图，显示所有项目的进度、状态、客户信息等。有一个看板，能够看到产生预警的事件

## 数据模型

### 角色定义 (role_definition)
系统预置9个角色，分为 admin / supervisor / executor 三类。
supervisor 角色可指派特定 executor 角色到项目。

### 人员指派 (project_assignment)
替代旧的 project_team 模型，支持：
- role_code 外键关联角色定义
- target_phase_seq 可选，指定该人员负责的具体工序（null 表示跨工序）

### 工序状态 (project_phase.status)
每道工序有独立的子状态流转，值从模板定义中读取：
- 机械设计: 未开始 → 设计中 → 图纸已下发
- 生产: 未开始 → 生产中 → 生产完成 → 已发货
- 调机: 未开始 → 安调中 → 安调完成
- 验收: 未开始 → 已验收
- 尾款: (收款进度替代)

### 子任务 (phase_subtask)
工序内可拆分为多个子任务，每个子任务有独立负责人和状态。
子任务与 incident 的区别：子任务是计划内的工作分解，incident 是计划外的意外事件。
